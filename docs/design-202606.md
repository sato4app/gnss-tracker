# GNSS Tracker 全体設計（202606）

`docs/RequiredSpec-202606.md` に基づく GNSS 受信・解析 PWA の実装設計。
Vanilla JS（ES モジュール）・フレームワーク/バンドラ不使用。Android Chrome 前提・BLE（Web Bluetooth）専用。

## 0. システム構成（確定）

```
[u-blox MAX-M10S] --UART(38400)--> [Raspberry Pi Pico W] --BLE(NUS notify)--> [Android Chrome / 本PWA]
```

- **Pico↔M10S の UART は 38400 baud**（旧 9600 から変更）。`micropython/main.py` は起動時に
  38400 で NMEA が来るか確認し、来なければ工場出荷時の 9600 で開いて UBX-CFG で 38400 へ切替・再確認する。
  9600 ではマルチGNSSのフル出力（特に GSV）が帯域に収まらず間引かれるため、38400 で全系統の GSV が完全に届く。
- **Pico↔Android は Bluetooth(BLE / Nordic UART Service) のみ**。WebSocket / WiFi は廃止（`config.py` も不要）。
  iPhone/iPad は Web Bluetooth 非対応のため対象外。
- **アプリ側の対応**: 当初から BLE 専用実装（WebSocket コード・トランスポート選択 UI は存在しない）のため、
  WS 廃止に伴うアプリ変更は不要。38400 化で増えた衛星数・データ量は、ストリーミング行復元
  （`line-buffer.js`）・全 GSV 集約（`epoch-assembler.js`）・`requestAnimationFrame` による再描画
  スロットリング（`app.js`）で吸収する。開発用モック（`mock-feeder.js`）も GPS/GLONASS/Galileo/BeiDou/QZSS
  の5系統フル GSV を生成し、実機相当のデータ量で解析表示を検証できる。

## 1. データフロー

```
[Pico W BLE(NUS notify)] ─┐
                          ├→ ble-client.js ─ onFrame(チャンク) ─┐
[mock-feeder.js（開発用）] ┘                                    │
                                                                ▼
                                       line-buffer.js（チャンク連結→ \n / *XX で行復元）
                                                                ▼
                                       nmea-parser.js（チェックサム検証＋GGA/RMC/GSA/GSV/VTG/GST）
                                                                ▼
                                       epoch-assembler.js（同一時刻群 → 1エポック確定）
                                                                ▼
                            onEpoch ──┬→ app.js（rAF スロットリングで再描画）
                                      │     ├→ map.js（現在地・精度円・軌跡）
                                      │     ├→ ライブ表示（fix/衛星数/DOP/精度）
                                      │     ├→ sky-plot.js / snr-chart.js
                                      │     └→ accuracy.js（水平精度推定）
                                      └→ recorder.js（スナップショット / 静的測位収集）
                                            ├→ accuracy.js（DRMS/CEP 等の集計）
                                            ├→ scatter-plot.js（散布図）
                                            ├→ storage.js（IndexedDB 保存）
                                            └→ exporter.js（CSV/GPX/JSON）
```

- 壊れた行（チェックサム不一致・分割送信失敗による欠損）はパーサで `valid:false` となり捨てる（件数は受信品質統計に計上）。接続は維持。
- 再描画は `requestAnimationFrame` で1フレーム1回にスロットリング。
- 行・エポックは `stream-stats.js`（受信品質統計）にも分岐する。Pico は 5 秒ごとに独自文
  `$PPICO`（UART受信行数などのカウンタ）を配信し、アプリが突合して各段の取りこぼしを検出する
  （`docs/rx-integrity-202607.md`）。`$PPICO` は統計のみでエポック組み立てには回さない。

## 2. モジュール責務

| ファイル | 責務 |
|---|---|
| `js/app.js` | エントリ。各モジュール結線・接続状態管理・エポック描画・タブ切替・記録UI・SW 登録 |
| `js/ble-client.js` | Web Bluetooth(NUS) 接続。`filters:[{name:'picow'}]`。`gattserverdisconnected` 監視・指数バックオフ自動再接続（手動切断と区別） |
| `js/line-buffer.js` | BLE チャンク（ATT_MTU−3 ごと分割）の連結と行復元 |
| `js/nmea-parser.js` | GGA/RMC/GSA/GSV/VTG/GST パース＋チェックサム検証。マルチ GNSS 系統判別 |
| `js/epoch-assembler.js` | 同一時刻センテンス群 → 1エポック（GSV の total/msg index 集約含む）。アイドルタイムアウトで最終エポックも確定。GSV 部分欠落（`gsvMissing`）も算出 |
| `js/stream-stats.js` | 受信品質統計。行/エポックの通過・破棄数と `$PPICO`（Pico側カウンタ）を突合し、M10S→Pico→BLE→アプリ各段の取りこぼしを計数・推定（`docs/rx-integrity-202607.md`）。静的測位1回分の差分を `summary.rxStats` に保存 |
| `js/accuracy.js` | 水平精度推定（GST→DRMS 優先 / HDOP×UERE フォールバック）、静的測位集計（平均/中央値/σ/DRMS/2DRMS/CEP50/CEP95/標高統計） |
| `js/map.js` | Leaflet＋地理院地図（標準/淡色/写真・既定 z=17）。現在地・水平精度円・軌跡・記録マーカー・追従 |
| `js/canvas-view.js` | Canvas 系ビュー共通基底（ctx 取得・DPR 対応リサイズ・リサイズ再描画）。sky/snr/scatter が継承。キャリーフォワード判定 `holdDecision`＋`HOLD_MS` を提供 |
| `js/sky-plot.js` | 極座標スカイプロット（半径=仰角、角度=方位、大きさ=SNR、色=系統、塗り=使用中）。衛星データ欠落時は前フレーム保持（`HOLD_MS`=8秒 失効でクリア。`docs/sat-view-hold-202607.md`） |
| `js/snr-chart.js` | 衛星別 SNR バー（系統色分け・使用中強調）。衛星データ欠落時は前フレーム保持（`HOLD_MS` 失効。同上） |
| `js/scatter-plot.js` | 静的測位の中心オフセット散布図（東西 m × 南北 m、CEP50/DRMS 円重畳） |
| `js/format.js` | 表示用フォーマッタ（純粋関数）。数値整形・HTML エスケープ・測位バッジ・記録テキスト |
| `js/recorder.js` | (A) スナップショット 1点保存 / (B) 静的測位 N秒 or Mエポック収集→集計。収束自動停止（最低時間＋中心・DRMS が10秒横ばいで停止。品質ゲート併用。`docs/static-autostop-202607.md`）。収集中ライブ統計。画面非表示で一時停止 |
| `js/storage.js` | IndexedDB ラッパ（`gnssDB`） |
| `js/exporter.js` | CSV（BOM 付き UTF-8）/ GPX / JSON（生エポック群＋集計値）出力 |
| `js/wake-lock.js` | Wake Lock 取得・解放・`visibilitychange` 再取得 |
| `js/tile-cache.js` | `tile_manifest.json`（Format A 座標リスト）の一括事前ダウンロード → Cache API（`gsi-tiles`） |
| `js/settings-ui.js` | 設定タブの UI 配線（測位パラメータ・地図種別・軌跡）。`settings` 更新＋永続化 |
| `js/tile-ui.js` | オフラインタイル DL の UI 配線（進捗・キャンセル・DL 済み枚数/version 表示）。取得は tile-cache.js |
| `js/mock-feeder.js` | 開発用擬似 NMEA 配信（GST あり、フレーム分断テスト付き）。実 BLE と同じ `onFrame` I/F |
| `sw.js` | アプリシェルのプリキャッシュ＋地理院タイルの cache-first（`gsi-tiles` を tile-cache.js と共有） |

## 3. エポック構造

```js
{
  t: Date,                  // RMC日付＋UTC時刻
  lat, lon, altMSL, geoidSep,
  fixQuality, fixMode, status,
  satsUsed, satsInView,
  pdop, hdop, vdop,
  satellites: [{ sys, prn, elev, azim, snr, used }],
  usedBySys, inViewBySys,   // 系統別内訳
  speedKmh, course,
  latStd, lonStd, altStd,   // GST があれば
}
```

## 4. IndexedDB スキーマ（`gnssDB` v1・端末内のみ）

- **`sessions`** (keyPath: `id`)
  `{ id, type: 'snapshot'|'static', label, memo, createdAt, endedAt?, summary }`
  - snapshot summary: `{ lat, lon, altMSL, fixQuality, hdop, satsUsed }`
  - static summary: `{ lat, lon, altMSL, count, drms, cep50, cep95, stopReason }`
    （`stopReason`: `converged`|`timeout`|`maxEpochs`|`manual`。旧データは無し=manual 相当）
- **`points`** (keyPath: `id`, index: `bySession`)
  - snapshot: `{ id, sessionId, kind:'snapshot', sample }`
  - static: `{ id, sessionId, kind:'static', stats(集計値＋散布図オフセット), samples(生エポック群) }`
- **`settings`** (keyPath: `key`)
  `uere` / `staticMaxSec` / `staticMaxEpochs` / `staticAutoStop` / `staticMinSec` / `mapType` / `trackEnabled`

## 5. 既定値（仕様 §8 の未確定事項への暫定回答）

| 項目 | 既定値 |
|---|---|
| GST | 出力があれば DRMS に使用。無ければ HDOP×UERE に自動フォールバック |
| 静的測位の収集条件 | 収束自動停止 ON（最低 30 秒 → 中心・DRMS が 10 秒横ばいで停止）。上限 60 秒 / 120 エポックはタイムアウト（0=無制限、設定タブで変更可） |
| 静的測位の地点名 | 開始前に入力ダイアログで確認。既定名 `yyyy-mm-dd-xx`（xx=01からの同日連番） |
| UERE | 5 m（設定タブで変更可） |
| アプリ名 | GNSS Tracker |
| 配置 | 本リポジトリ直下（新規構成） |

## 6. オフライン設計

- アプリシェル（HTML/CSS/JS/Leaflet/アイコン）は SW インストール時にプリキャッシュ。
- 地理院タイルは閲覧時に cache-first で `gsi-tiles` キャッシュへ。設定タブの「タイル事前ダウンロード」で `data/tile_manifest.json` 記載分を一括取得（中止可・進捗表示・キャッシュ済はスキップ・同時4接続）。
- **タイルマニフェストは `data/tile_manifest.json` を唯一の置き場とする**（ルート直下に複製は置かない）。`tile-cache.js` はこの 1 ファイルを直接読む。`data/tile_manifest.json` の層構造（`layers.<名前>.{z, tiles:[[x,y]]}`）は `tile-cache.js` のローダ／ダウンローダとそのまま互換。
- BLE 受信・解析・記録・エクスポートはネットワーク不要で完全動作。

## 7. 開発・確認

```bash
# ローカルサーバ（ES モジュールのため必須）
python -m http.server 8000
# → http://localhost:8000 を開き、設定タブ「モックNMEA配信」を ON

# 構文チェック＋純粋ロジックの簡易テスト
node tools/check-syntax.mjs
```

実機（Android Chrome）は HTTPS 配信（Vercel 等）でアクセスし、「接続」→ `picow` を選択する。
